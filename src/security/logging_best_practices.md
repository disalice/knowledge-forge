## 概要

アプリケーションのログはプレーンテキストではなく、JSONなどの構造化フォーマットで出力し、機密データがログに混入しないようマスキング（難読化）を必須とする。

## なぜ必要なのか

- **構造化:** DatadogやCloudWatch Logsなどのログ監視ツールでの検索性・集計・アラート設定を容易にするため。
- **マスキング:** ログシステムへの権限を持つ開発者や運用者が、ユーザーの機密情報にアクセスできてしまうセキュリティリスク（コンプライアンス違反）を防ぐため。

## 実装標準

- ロギングライブラリ（Winston, Zap, Logbackなど）の設定で、出力をJSONフォーマットに統一する。
- ミドルウェアやインターセプター層で、リクエスト/レスポンスのボディやヘッダをログ出力する前に、特定のキー（`password`, `token`, `credit_card`, `email` など）を `***` に置換するフィルタを噛ませる。

## 設計・実装時のチェックリスト

- [ ] ログフォーマットはパースしやすいJSONになっているか？
- [ ] リクエスト・レスポンスをロギングする際、個人情報や認証情報が平文で出力されていないか？
- [ ] Trace ID / Request ID がログに含まれており、リクエスト全体をトレースできるか？

## アンチパターン

- `console.log("User updated: " + JSON.stringify(user))` のように、テキストとJSONが混ざった文字列を出力する。
- エラー発生時に、リクエストオブジェクト全体をそのままログにダンプする（パスワードなどが漏洩する原因の第1位）。

## 実施・導入によるトレードオフ

- マスキング処理（特に巨大なJSONの深い階層の走査）によるCPUオーバーヘッド。
- ローカル開発環境でのログの視認性低下（Pretty-printツールを併用することで解決可能）。

## 適用範囲と例外

すべてのアプリケーションログ、アクセスログ。

## 検証方法

- 個人情報を含むダミーリクエストを送信し、出力されたログファイル・ストリーム内で対象文字列がマスキングされているかを目視またはテストコードでアサートする。

## 概要
OIDC（OpenID Connect）は、AWSやGCPなどのクラウドリソースへのセキュアなアクセスを実現するためのプロトコルです。CI/CDパイプライン（例: GitHub Actions）やアプリケーションからクラウドリソースにアクセスする際、永続的な静的クレデンシャル（アクセスキーやシークレットキー）の発行・利用を禁止し、一時的な認証情報（動的クレデンシャル）の取得を標準とすることが推奨されます。これにより、セキュリティリスクを大幅に低減できます。

## なぜ必要なのか
静的なアクセスキーが漏洩した場合、攻撃者は長期間にわたってインフラを悪用するリスクがあります。例えば、暗号資産のマイニングや機密データの削除などが行われる可能性があります。OIDCを使用することで、パイプラインの実行時など「必要な瞬間」だけ有効な一時トークンが発行され、認証情報の漏洩リスクを劇的に低減できます。

## 実装標準
1. **Trust関係の構築:** CI/CDツール（GitHub Actionsなど）とクラウドプロバイダ（AWS IAMのWeb IDフェデレーションやGCPのWorkload Identityなど）の間でOIDC連携を設定します。
2. **権限の絞り込み:** クラウド側の信頼ポリシー（Trust Policy）で、リポジトリ名やブランチ名（例: `main` ブランチのみ）を指定し、想定外の場所からロールを引き受けられないように制限します。
3. **動的クレデンシャルの取得:** CI/CDのステップ内で一時的なSTSトークンを取得し、それを用いてクラウドリソース（S3へのデプロイやコンテナレジストリへのPush）へアクセスします。

## 設計・実装時のチェックリスト
- [ ] IAMユーザーのアクセスキー（静的認証情報）が新規に発行されていないか？
- [ ] GitHub ActionsなどのCI/CDパイプラインに、クラウドのアクセスキーがSecretsとして登録されていないか？
- [ ] クラウド側のIAMロールの信頼ポリシーで、アクセス元のリポジトリやブランチが厳密に制限されているか？

## アンチパターン
- **IAMユーザーの使い回し:** アプリケーション、CI/CD、開発者全員が、同じ強力な権限を持ったIAMユーザーのアクセスキーを使い回すこと。
- **広すぎる信頼ポリシー:** OIDCの信頼ポリシーでリポジトリ名を指定せず、組織内のどのリポジトリからでもデプロイ権限を持つロールを引き受けることができてしまう状態。

## 実施・導入によるトレードオフ
初期のOIDC連携設定や、IAMロールの権限設定（Trust Policyの正しい記述）に関する学習コストと構築の手間がかかります。また、ローカル環境での開発時にも、静的キーではなくSSOを用いた一時認証情報（例: `aws sso login`）を利用する仕組みを整える必要があります。

## 適用範囲と例外
- **スコープ:** クラウド環境へのデプロイパイプライン、およびクラウド上のリソースにアクセスするすべてのアプリケーション。
- **例外:** OIDC連携をシステム上サポートしていない、レガシーな外部SaaSツールからのアクセス（この場合はアクセスキーの定期的なローテーションを必須とします）。

## 検証方法
CI/CDパイプラインを実行し、ソースコードやCIのSecretsに静的クレデンシャルを持たせない状態で、クラウドリソースへのアクセス（デプロイ等）が正常に完了することを確認します。

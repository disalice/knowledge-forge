## 概要
シークレット情報の適切な管理は、アプリケーションのセキュリティを確保するために不可欠です。APIキー、データベースのパスワード、暗号化キー、署名用シークレットなどの機密情報をソースコードやGitリポジトリにハードコードすることを禁止し、これらの情報はAWS Secrets Manager、HashiCorp Vault、または安全に管理された環境変数を通じて、実行時にアプリケーションへ動的に注入する設計を推奨します。

## なぜ必要なのか
シークレット情報が一度でもGitの履歴に残ると、リポジトリへのアクセス権を持つすべての人間や、万が一ソースコードが流出した際の悪意ある攻撃者に認証情報が渡ってしまいます。これは、インフラの不正利用や顧客データの漏洩といった、致命的なセキュリティインシデントに直接繋がるため、適切な管理が必要です。

## 実装標準
1. **ローカル環境:** `.env` ファイル等のローカル専用設定ファイルを使用し、このファイルは必ず `.gitignore` に追加してコミット対象外とします。
2. **CI/CD環境:** GitHub ActionsのSecrets機能やCIツールの機密情報管理機能を利用し、スクリプト内に平文で書かないようにします。
3. **本番・検証環境:** クラウドプロバイダのシークレット管理サービス（AWS Secrets Manager, GCP Secret Manager等）を利用し、アプリケーション起動時やコンテナ展開時に環境変数として安全に注入します。

## 設計・実装時のチェックリスト
- [ ] 開発用のダミー値であっても、本番用と同じファイルやクラスにシークレット情報をベタ書きしていないか？
- [ ] プロジェクトの `.gitignore` に `.env` や `credentials.json` などの機密ファイルが確実に登録されているか？
- [ ] クラウド上のシークレット情報（Secrets Manager等）にアクセスできるIAM権限が、本当に必要なサービスや少人数の管理者に最小特権の原則で絞られているか？

## アンチパターン
- **設定ファイルへの直接記述:** 環境ごとの設定ファイル（例: `config.production.json`）に本番用のAPIキーをそのまま書き込み、リポジトリにプッシュしてしまう。
- **Base64エンコードによる偽装:** Base64等でエンコードしただけの文字列を「暗号化されているから安全」と誤認し、ソースコードに埋め込む（Base64は暗号化ではなく単なるエンコーディングであり、誰でも即座に復元可能です）。

## 実施・導入によるトレードオフ
シークレット管理ツールのインフラリソース費用がわずかに発生するほか、ローカル開発環境のセットアップ時に「環境変数の共有と設定」というひと手間がオンボーディングコストとして追加されます。

## 適用範囲と例外
- **スコープ:** すべてのアプリケーションコード、インフラ構成コード（Terraform等）、および運用スクリプト
- **例外:** なし

## 検証方法
- **予防:** 開発者のローカル環境に `git-secrets` や `Talisman` などの pre-commit フックを導入し、コミット時にシークレットを検知してブロックします。
- **検知:** CIパイプラインに `TruffleHog` や `Gitleaks` などのシークレットスキャンツールを組み込み、リポジトリ内にハードコードされた認証情報が存在しないことをPRの都度、自動検証します。

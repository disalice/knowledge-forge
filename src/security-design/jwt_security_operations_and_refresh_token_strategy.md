## 概要
JWT（JSON Web Token）は、APIの認証においてトークン自体にセッション状態を持たせるステートレスな認証を実現します。このナレッジでは、JWTを安全に運用するための基準と、リフレッシュトークン戦略について説明します。特に、アクセストークンの有効期限を短く設定し、リフレッシュトークンを使用してトークンの再発行を管理することが重要です。

## なぜ必要なのか
- **不正アクセスの期間短縮:** JWTは一度発行されると、サーバー側から強制的に無効化することができません。有効期限が長いJWTが漏洩した場合、攻撃者に不正アクセスを許し続けるリスクがあります。
- **安全なセッション維持:** 短寿命のアクセストークンと長寿命のリフレッシュトークンを分離し、適切なリフレッシュ機構を設けることで、ユーザー体験とセキュリティを両立させます。

## 実装標準
1. **アクセストークン（Access Token）の設計:**
   - 有効期限（`exp` クレーム）を10〜15分程度に設定。
   - 署名アルゴリズムには非対称暗号（RS256やEdDSA）を採用し、APIサーバー側は公開鍵のみで検証できるようにする。

2. **リフレッシュトークン（Refresh Token）の設計:**
   - 有効期限を数日〜数週間に設定。
   - トークンの一意なID（`jti` クレーム）をデータベースやKVS（Redisなど）に保存し、サーバー側で個別に無効化できるようにする。

3. **トークンローテーション（Refresh Token Rotation）の導入:**
   - クライアントがリフレッシュトークンを使用して新しいアクセストークンを要求した際に、新しいリフレッシュトークンも発行し、古いリフレッシュトークンを無効化する。

## 設計・実装時のチェックリスト
- [ ] アクセストークンの有効期限（exp）が30分以上に設定されていないか？
- [ ] JWTの署名検証ロジックで、アルゴリズムの偽装を防ぐために `alg: "none"` や対称暗号（HS256）への強制書き換えを拒否しているか？
- [ ] JWTを `LocalStorage` ではなく、`HttpOnly`・`Secure`・`SameSite=Lax/Strict` 属性が付与されたCookieに格納しているか？
- [ ] ログアウトAPIが呼び出された際、リフレッシュトークンが確実に破棄・無効化されるよう実装されているか？

## アンチパターン
- **アクセストークンのみの長期運用:** 有効期限が「30日間」のアクセストークンを発行し、データベースへのアクセスを減らす設計。トークンが漏洩した際の被害が甚大。
- **LocalStorageへの格納:** `LocalStorage` にJWTを保存し、XSS攻撃によって認証情報が盗まれるリスク。
- **検証なしのデコード:** JWTの署名を検証せずにデコードして処理を行う実装。

## 実施・導入によるトレードオフ
- リフレッシュトークンの状態管理やローテーションの判定を行うため、サーバー側に一部「ステートフル」な管理層が必要になり、完全なステートレスのメリットが微減します。
- クライアント側でのアクセストークンの期限切れを検知し、自動リフレッシュを行うロジックの実装が複雑になります。

## 適用範囲と例外
- **スコープ:** Webフロントエンド、モバイルアプリ、ユーザー認証が必要なすべてのパブリックRESTful / HTTP API。
- **例外:** サーバー間の同期通信など、OAuth 2.0のクライアントクレデンシャルズグラントを用いるクローズドな環境。

## 検証方法
- ログイン処理を実行し、発行されたJWTをデコードツールで確認し、`exp`が15分以内であること、`alg`が非対称暗号であることを確認します。
- テストコードで、有効期限の切れたアクセストークンでAPIを呼び出した際に `401 Unauthorized` が返ることを検証します。
- 古いリフレッシュトークンを再送信し、サーバー側で認証エラーが発生することを確認します。

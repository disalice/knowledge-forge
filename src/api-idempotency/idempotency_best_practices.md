## 概要

リソースの作成（POST）や更新（PUT/PATCH）を行うAPIにおいて、同一のリクエストがネットワークエラー等により複数回送信された場合でも、システムの状態が1回だけ処理されたのと同じになるよう「冪等性（Idempotency）」を担保することが推奨されます。

## なぜ必要なのか

クライアント（フロントエンドやモバイルアプリ、他マイクロサービス）は、タイムアウト発生時に安全にリトライを行う必要があります。冪等性が担保されていないと、二重決済やリソースの重複作成といった重大な不整合が発生する可能性があります。

## 実装標準

- クライアントから一意なキー（`Idempotency-Key` ヘッダなど）を送信させる。
- バックエンド側では、受け取ったキーと処理結果（または処理中ステータス）をKVS（Redis等）やDBに一定期間保存する。
- 同一キーのリクエストが来た場合、再処理を行わず、保存してある前回と同じ結果を返す。

## 設計・実装時のチェックリスト

- [ ] POST/PATCHメソッドを使用するAPIで、リトライ時の影響を考慮しているか？
- [ ] 冪等性キーを保存・参照する仕組み（RedisやRDBのユニーク制約）が設計に組み込まれているか？
- [ ] 同時実行（同一キーがほぼ同時に到達した際）の排他制御（ロック）が考慮されているか？

## アンチパターン

- 単にDBへのINSERTを連続で行い、ユニーク制約エラーが出たら `500 Internal Server Error` を返す（クライアントがリトライ成功か判別できない）。
- `GET` や `DELETE` など、本来冪等であるべきHTTPメソッドの仕様を破壊し、呼ぶたびに状態が変わる実装になっている。

## 実施・導入によるトレードオフ

- リクエストごとのキー保存・検証によるレイテンシの微増。
- KVSやDBへの一時データのストレージコスト増加。
- 実装とテストの複雑化。

## スコープと例外

- **スコープ:** 決済、ポイント付与、注文、メール送信など、重複実行が致命的なビジネスロジックを含むAPI。
- **例外:** ログの単なる追記や、重複しても実害がない（またはシステム側で定期的にクリーンアップされる）軽微なデータ作成。

## 検証方法

- 同一の `Idempotency-Key` を用いて連続でAPIをコールし、1回目のみ処理が走り、2回目以降はキャッシュされた正常レスポンスが返るか（DBが増殖していないか）をテストする。
- 意図的にレイテンシを発生させ、同時に同じリクエストを投げた場合のレースコンディションを検証する。

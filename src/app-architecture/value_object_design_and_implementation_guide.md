## 概要
クリーンアーキテクチャやドメイン駆動設計（DDD）において、金額、メールアドレス、電話番号、各種IDなどのドメインの概念を、標準のプリミティブ型（`String` や `Int` など）でそのまま引き回すのではなく、ルールと振る舞いをカプセル化した不変（Immutable）なクラス（値オブジェクト）として定義することが設計標準となります。

## なぜ必要なのか
- **不正な状態の排除:** 「マイナスの金額」や「@が含まれないメールアドレス」といった不正な値が、システム（ユースケース層やドメイン層）に侵入することをインスタンス生成の時点で防ぐため。
- **ロジックの散財防止:** バリデーションや値に関する計算ロジック（例: 金額の足し算、通貨のチェック）がコードベースの各所に散らばるのを防ぎ、一つのクラスに凝集させるため。
- **シグネチャの明確化:** `updateUser(String, String, String)` のような引数の順番間違いによるバグを、`updateUser(UserId, Email, PhoneNumber)` のように型で防ぐため。

## 実装標準
以下のルールに従って値オブジェクトを実装します。
1. **完全コンストラクタ:** インスタンス生成時（コンストラクタまたはファクトリメソッド）に必ずバリデーションを行い、不正な値の場合は例外（`IllegalArgumentException` など）をスローする。
2. **不変性（Immutability）:** 生成されたインスタンスの状態は絶対に変更できないようにする（セッターを持たせない）。値を変更したい場合は、新しいインスタンスを生成して返す。
3. **等価性（Equality）:** インスタンスの参照（メモリ上のアドレス）ではなく、保持している「属性の値」がすべて同じであれば同一とみなすよう、`equals` メソッド等をオーバーライドする。

## 設計・実装時のチェックリスト
- [ ] その値に関するバリデーションルールは、コントローラー層などではなく値オブジェクト内に定義されているか？
- [ ] インスタンス生成後に値を変更するセッターメソッドが存在していないか（不変性が保たれているか）？
- [ ] DBのIDなどを単なる `int` や `String` ではなく、`UserId` や `OrderId` のような型として扱っているか？

## アンチパターン
- **プリミティブ型への執着 (Primitive Obsession):** ドメイン固有の概念であるにもかかわらず、すべてを `String` や `Integer` で表現し、使うたびにバリデーションを行っている状態。
- **インフラへの依存:** 値オブジェクトの中にデータベースへの問い合わせ（Repositoryの呼び出し）や、外部APIへの通信ロジックを混ぜてしまうこと（値オブジェクトは純粋なドメイン知識のみを持つべきです）。

## 実施・導入によるトレードオフ
- クラス（ファイル）の数が純粋に増加します。
- 単純なCRUDしか行わない極小規模なアプリケーションでは、クラス定義とマッピングのオーバーヘッドがメリットを上回る（過剰設計になる）可能性があります。

## 適用範囲と例外
- **スコープ:** クリーンアーキテクチャにおけるエンティティ層（ドメイン層）、およびユースケース層。
- **例外:** クライアントと通信するAPIのレスポンスやリクエストボディ（DTO）、DBとやり取りするDAOモデルなど、フレームワークやライブラリの制約上プリミティブ型や可変（Mutable）オブジェクトが求められる境界部分。

## 検証方法
値オブジェクトに対するユニットテスト（単体テスト）を作成し、「不正な値で初期化しようとした時に正しく例外がスローされること」「値の計算やフォーマット出力が正しいこと」を検証する。外部モックが不要なため、非常に高速で安定したテストが書けます。

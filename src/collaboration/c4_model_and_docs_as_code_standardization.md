## 概要
C4モデルは、システムの全体像やコンポーネント間の依存関係を視覚的に表現するためのフレームワークです。このモデルを用いることで、異なる抽象度の図を階層化し、開発者間のコミュニケーションを円滑にします。また、Docs as Codeのアプローチを採用し、MermaidやPlantUMLなどのツールを使用して図をコードとして管理することで、バージョン管理やメンテナンスを容易にします。

## なぜ必要なのか
- **曖昧さの排除:** 開発者ごとに異なる図を描くことで生じるコミュニケーションの齟齬を防ぎます。
- **認知負荷の低減:** 抽象度を分けることで、ビジネス側や新人エンジニアがそれぞれの目的に応じてシステムを理解しやすくします。
- **メンテナンス性の向上:** コードとして図を管理することで、更新が容易になり、陳腐化を防ぎます。

## 実装標準
C4モデルに基づく図の作成は、以下の階層（ズームレベル）を意識して行います。通常、上位2つのレベルを描くだけでも効果があります。
1. **Context（システムコンテキスト）:** ユーザーと対象システム、外部システムとの関係を描く。
2. **Containers（コンテナ）:** 対象システム内部の独立してデプロイ可能な単位を描く。
3. **Components（コンポーネント）:** コンテナ内部の主要なモジュールや機能境界を描く。
4. **Code（コード）:** クラス図など（手動での作成は推奨しません）。

## 設計・実装時のチェックリスト
- [ ] 異なる抽象度の要素が混在していないか？
- [ ] 図の中の矢印が何を表しているかが明記されているか？
- [ ] 図はテキストベースで書かれ、Markdownファイル内で直接レンダリング・編集できる状態になっているか？

## アンチパターン
- **神の図（God Diagram）:** システムのすべてを1枚の図に押し込むことで、視覚的に混乱を招く。
- **説明のない図形と線:** 意味が不明な図形や線が多用され、意図が伝わらない図。

## 実施・導入によるトレードオフ
- C4モデルやMermaidの学習に初期コストがかかる。
- システムの実装変更に伴い、アーキテクチャ図も更新する運用コストが発生する。

## 適用範囲と例外
- **スコープ:** Design Doc（RFC）、ADR、リポジトリの `README.md`、新規参画者のオンボーディング資料。
- **例外:** 一時的なブレインストーミングやプロトタイプ開発では厳密なルールは不要。

## 検証方法
新しくチームに参画したメンバーに「Context」および「Containers」レベルの図を見せ、システム全体の構成やユーザーの関わり方を5分程度で説明できるかを確認します。

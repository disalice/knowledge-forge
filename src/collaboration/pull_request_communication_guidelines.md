## 概要

プルリクエスト（PR）におけるコミュニケーションのガイドラインは、ソフトウェア開発においてコードの変更をチームでレビューし、協力して品質を向上させるための重要な要素です。PRの作成、フィードバックの提供、フィードバックへの対応に関する具体的な手法をまとめることで、チーム内の文脈を共有し、建設的な議論を促進します。

## なぜ必要なのか

- **文化の維持と育成:** 組織やプロジェクトが変化・成長しても、チームが目指す文化や協調性を保つため。
- **文脈の共有:** チーム内の誰が読んでもコード変更の意図を理解できるようにするため。
- **ネガティブバイアスの排除:** オンラインでのテキストコミュニケーションはネガティブに受け取られがちであるため、これを防ぎ建設的な議論を行うため。
- **スキルの向上:** グループによる批評を通じて、プロフェッショナルなスキル、チームの知識、プロダクトの品質を向上させるため。

## 実装標準

以下の3つのフェーズにおいて、具体的なコミュニケーション手法を取り入れます。

1. **PRの作成時:**
   - 変更の目的（What）と背景（Why）を明記し、過去の経緯を知らない人でも理解できるようにする。
   - 求めるフィードバックの種類を明確にする。
   - 作業中（WIP）の場合は、タイトルに `[WIP]` をつけ、フィードバックが欲しいタイミングを明示する。
   - 特定の個人やチームを `@mention` する際は、メンションの理由を添える。

2. **フィードバックの提供時:**
   - Issueのコンテキストを理解する。
   - 「命令」ではなく「質問・提案」の形をとる。
   - コード変更の理由を明確に説明し、改善案を提示する。
   - 謙虚な姿勢を保ち、ポジティブな言葉や絵文字を活用する。

3. **フィードバックへの対応時:**
   - 賛否両論のフィードバックには感謝の意から始める。
   - すべてのコメントに対応し、解決した場合はフォローアップを行う。
   - 理解できない場合は明確化を求める。

## 設計・実装時のチェックリスト

- [ ] PRの目的と背景が明記されているか？
- [ ] どのようなフィードバックが欲しいか、タイミングが明記されているか？
- [ ] `@mention` した相手に対して、メンションした理由を伝えているか？
- [ ] 強い反対意見がある場合、冷静に考える時間を取ったか？
- [ ] 指示ではなく「質問」や「提案」でコメントしているか？
- [ ] ネガティブバイアスを意識し、ポジティブな言葉や絵文字でトーンを調整しているか？
- [ ] 指摘に対する感謝から入り、理由をわかりやすく説明しているか？

## アンチパターン

- **暗黙の了解を前提とする:** 背景や歴史を全員が知っていると思い込み、説明を省くこと。
- **感情的な反応と強い言葉:** 侮蔑的な言葉や過激な表現を使うこと。
- **テキストでの議論に固執する:** 混乱や議論が白熱しているにもかかわらず、テキストでのやり取りを続けること。

## 実施・導入によるトレードオフ

- コメントのトーンや言い回しに気を配るため、レビューを書くのに少し時間がかかる場合がある。
- PR作成時に丁寧な説明を書くための初期コストがかかる。

## 適用範囲と例外

- **スコープ:** すべてのPull Requestにおけるチーム内外のコミュニケーション。
- **例外:** PR上でのテキストコミュニケーションで混乱や対立が深まった場合は、テキストでのやり取りを中止し、同期的な会話に切り替える。

## 検証方法

- PRのコメント欄が建設的な議論の場になっているかを定期的に確認する。
- コードレビュー後に、メンバーが不当な攻撃を受けたと感じていないかを確認する。

## 概要
大量データのリストを取得するAPIにおいて、`OFFSET` と `LIMIT` の組み合わせを避け、カーソルを利用したページネーションを採用することを提案します。

## なぜ必要なのか
`OFFSET` を使用すると、指定された行数分のデータをデータベースから読み込んでから破棄するため、ページが後ろに進むにつれてパフォーマンスが低下します。また、データの追加や削除が行われると、要素の抜け漏れや重複表示が発生しやすくなります。

## 実装標準
- APIレスポンスに `next_cursor`（エンコードされた最後の要素のIDやタイムスタンプ等）を含める。
- 次回リクエスト時に `next_cursor` を条件としてクエリを実行する（例: `WHERE id > ? LIMIT ?`）。

## 設計・実装時のチェックリスト
- [ ] ソート条件に使用するキーの組み合わせが一意（ユニーク）になっているか
- [ ] ソート条件に対して適切にDBインデックスが張られているか
- [ ] クライアントが次ページを取得するためのカーソル値がレスポンスに含まれているか

## アンチパターン
- ソートキーに重複する値（例：作成日時のみ）を使用し、クエリ結果の順序が非決定になってしまうこと。

## 実施・導入によるトレードオフ
- 特定のページ番号への直接ジャンプ機能（例: 「5ページ目へ飛ぶ」）が実装困難になる。

## 適用範囲と例外
- スコープ
  - 無限スクロールのUIを持つ画面のAPI
  - 将来的に数万件以上のデータが存在し得るテーブルのリストAPI
- 例外
  - データ件数が少なく、ユーザーが任意のページ番号を指定する必要がある管理画面などは `OFFSET` / `LIMIT` でも可。

## 検証方法
- 数百万件のテストデータを投入した状態で、末尾付近のデータ取得（ページネーション）のレスポンス速度を測定し、遅延がないことを確認する。

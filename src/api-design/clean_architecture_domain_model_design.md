## 概要
クリーンアーキテクチャやドメイン駆動設計（DDD）において、ビジネスロジック（計算、状態遷移のルール、バリデーション）はユースケース層に記述するのではなく、ドメインモデル（エンティティや値オブジェクト）自身の振る舞いとしてカプセル化することが設計基準となります。このアプローチにより、システムの保守性やテストの容易性が向上します。

## なぜ必要なのか
- **ロジックの散逸と重複を防ぐため:** ユースケース層にビジネスルールを記述すると、異なるユースケースで同じエンティティを扱う際にロジックの重複が発生し、仕様変更時の修正漏れの原因となります。
- **テストの容易性:** ユースケース層で複雑なビジネスロジックをテストする際、外部依存が多くなるためモックの準備が膨大になります。ドメインモデル内にロジックを閉じ込めることで、外部依存のない安定したユニットテストが可能になります。

## 実装標準
ユースケース層の責務を「ドメインモデルのオーケストレーション」に限定し、以下の4ステップを原則とします。
1. **取得:** リポジトリ経由でデータベースからエンティティを取得する。
2. **実行:** 取得したエンティティのメソッドを呼び出し、状態を変更させる。
3. **保存:** 状態が変更されたエンティティをリポジトリ経由で保存する。
4. **返却:** 結果をプレゼンテーション層に返す。

## 設計・実装時のチェックリスト
- [ ] エンティティが単なるデータの入れ物になっていないか？
- [ ] ユースケースクラスに複雑なビジネスロジックが書かれていないか？
- [ ] エンティティの状態を変更する際、ユースケース側で計算して `Setter` を使っていないか？

## アンチパターン
- **トランザクションスクリプト:** エンティティがデータ保持クラスになり、ユースケースクラスが肥大化し、すべてのビジネスロジックを担う状態。
- **ユースケース内でのドメインルールの判定:** ビジネスルールをユースケース内に書くのではなく、エンティティのメソッドに持たせるべきです。

## 実施・導入によるトレードオフ
従来のMVCフレームワークの開発スタイルに慣れた開発者にとって、オブジェクト指向に基づくパラダイムシフトと学習コストが必要です。

## 適用範囲と例外
- **スコープ:** クリーンアーキテクチャやDDDを採用しているシステムのドメイン層およびユースケース層全体。
- **例外:** 単純なCRUDの管理画面APIやCQRSアーキテクチャにおける「Readモデル」の構築処理では、複雑なドメインモデルを経由するオーバーヘッドを避けるため、直接SQLやクエリビルダを利用することを許容します。

## 検証方法
コードレビュー時にユースケース層のクラスを確認し、ビジネス要件に関する条件分岐が存在しないか、リポジトリとエンティティのメソッドを繋いでいるだけかを確認します。

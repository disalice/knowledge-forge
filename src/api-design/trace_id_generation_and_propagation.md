## 概要
システムの入口（API Gatewayなど）で一意のTrace IDを生成し、ログ出力時や外部システムへのリクエスト時に必ず引き回すことで、リクエストのトレーシングを行う手法を提案します。

## なぜ必要なのか
障害発生時やパフォーマンス低下時に、複数コンポーネント（API、DB、外部サービス）を跨いだリクエストの全容を素早く把握し、ボトルネックやエラー箇所を特定するために必要です。

## 実装標準
- W3C Trace Contextなどの標準仕様に則り、ヘッダ経由でTrace IDを伝播させる。
- アプリケーション側ではMDC (Mapped Diagnostic Context)を利用し、各スレッドで出力される全ログに自動でTrace IDが付与されるように基盤を設定します。

## 設計・実装時のチェックリスト
- [ ] システムの入口（最もエッジに近い層）でTrace IDが生成されているか
- [ ] すべてのアプリケーションログにTrace IDが自動付与されているか
- [ ] 非同期処理（メッセージキューやバックグラウンドワーカー）にもIDが引き継がれているか

## アンチパターン
- サービス間の通信で独自のヘッダ名を使ったり、伝播を途切らせてしまうこと。
- Trace IDをアプリケーションのビジネスロジック内で直接操作しようとすること。

## 実施・導入によるトレードオフ
- ログのデータ量とヘッダサイズの微増が発生します。
- 導入時の基盤・フレームワーク整備コストがかかる可能性があります。

## 適用範囲と例外
- スコープ: 外部からのすべてのAPIリクエストおよび、そこから派生するバックグラウンドジョブすべて。

## 検証方法
エンドツーエンドのテスト実行後、ログ検索基盤（Datadog, Kibana, CloudWatch Logs等）で1つのTrace IDでシステムの全ログが追えるか確認します。

## 概要
API設計において、サービスレベル指標（SLI）を正確に計測することは、システムの健全性を評価するために不可欠です。特に、REDメトリクス（Rate: リクエスト数、Errors: エラー数、Duration: 処理時間）を統一フォーマットで収集することで、可用性のSLO（サービスレベル目標）評価を行うことができます。このプロセスでは、クライアント起因のエラー（4xx）とサーバー起因のエラー（5xx）を明確に分離したエラーハンドリングを実装することが重要です。

## なぜ必要なのか
システムの健全性を「実際のユーザー体験」を基に客観的に評価するためには、エラーの種類を適切に分ける必要があります。ステータスコードが適切に分離されていないと、ユーザーの入力ミス（バリデーションエラーなど）が可用性SLIを低下させ、システム本来の障害検知やエラーバジェットの運用が形骸化してしまいます。

## 実装標準
1. アプリケーションのミドルウェア（またはインターセプター）層で、リクエストの終了時にアクセスログやメトリクスを自動出力する共通基盤を設ける。
2. 出力するメトリクスには、以下の情報を含めること。
   - タイムスタンプ
   - エンドポイントのルーティングパターン（例: `/users/{id}`）
   - HTTPステータスコード
   - 処理時間（レイテンシ）
3. 5xx系エラーのみを「サーバーの可用性低下（SLIの失敗）」としてカウントする集計ルールをダッシュボード側に設定する。

## 設計・実装時のチェックリスト
- [ ] バリデーションエラーや認証・認可エラーに対して、誤って500系エラーを返していないか
- [ ] 意図しない例外（未捕捉のException）が、確実に500番台として処理・記録される仕組みになっているか
- [ ] メトリクスのラベル（パス情報など）に、動的なID（`/users/123` など）が直接記録されていないか（カーディナリティの爆発を防ぐため、`/users/{id}` のように丸めること）

## アンチパターン
- メトリクスのパス情報に動的な値を含めてしまい、時系列データベース（PrometheusやDatadogなど）のメモリを枯渇させる（高カーディナリティ問題）。
- エラーの種別（4xx/5xx）を問わず、一律でSLI低下として扱い、無用なアラートを頻発させる（アラート疲労）。
- ビジネスロジックの各所に手動で計測用のコードを埋め込んでいる（共通化されていない）。

## 適用範囲と例外
- ユーザーや外部システムからのリクエストを受け付けるすべてのAPIサーバーに適用する。
- ユーザー影響のない非同期のバッチ処理などは、別の指標（スループットやジョブ完了時間など）を用いるため、通常のAPI用SLOとは分けて管理する。

## 検証方法
- 意図的にバリデーションエラー（400）とシステムエラー（500）を発生させるテストを行い、ダッシュボード上で500エラーのみが可用性SLIを低下させていることを確認する。
- パスパラメータを含むAPIをコールし、メトリクス基盤上でパスがテンプレート文字列（`{id}`等）として集約されているか確認する。

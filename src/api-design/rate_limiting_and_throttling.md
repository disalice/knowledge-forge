## 概要
APIのレート制限（Rate Limiting）とスロットリングは、公開されているAPIエンドポイントに対して、一定時間内に受け付けるリクエスト数の上限を設定する手法です。この制限を設けることで、悪意のある攻撃や過剰なリクエストからシステムを保護し、リソースの公平な利用を促進します。具体的には、上限を超過したリクエストに対しては、HTTPステータスコード `429 Too Many Requests` を返却し、再試行の目安となる `Retry-After` ヘッダを付与します。

## なぜ必要なのか
- **リソースの保護（DoS/DDoS対策）:** 悪意のある攻撃やバグによる無限ループから、データベースやバックエンドのリソース枯渇を防ぎ、システム全体のダウンを回避します。
- **テナント間の公平性担保:** 特定のユーザーやクライアントがリソースを独占することを防ぎ、他の正常なユーザーへのレスポンス低下を防ぎます。
- **セキュリティの向上:** ブルートフォース攻撃やクレドスタッフィング攻撃のリスクを軽減します。

## 実装標準
1. **実装レイヤーの選定:** アプリケーション層のミドルウェアやAPI Gateway、CDN/WAFでの実装を選定します。インフラ層での実装がパフォーマンス上は最適です。
2. **アルゴリズムの選定:** トークンバケットやリーキーバケットなどのアルゴリズムを採用します。
3. **キーの単位:** 「ユーザーIDごと」「IPアドレスごと」「APIキーごと」など、エンドポイントの性質に合わせてカウントのキーを設計します。
4. **レスポンス仕様:** 上限に達した場合は `429 Too Many Requests` を返し、可能であれば `Retry-After` ヘッダを付与します。

## 設計・実装時のチェックリスト
- [ ] 制限を超過した際、`500 Internal Server Error` ではなく `429 Too Many Requests` を返しているか？
- [ ] マルチサーバー・複数ポッド環境において、制限のカウント状態が共有されているか？
- [ ] 認証済みAPIと未認証APIで、適切なカウント基準が設定されているか？
- [ ] クライアントが適切にハンドリングできるよう、`Retry-After` や `X-RateLimit-Limit` などのヘッダを返却しているか？

## アンチパターン
- **単一インスタンスのメモリでのカウント:** オートスケール環境で制限のカウントがバラバラになる。
- **全APIで一律の制限値:** パフォーマンス特性に応じたチューニングがない。
- **リミット超過のサイレント破棄:** クライアントにエラーを返さず、無言でコネクションを切断する。

## 実施・導入によるトレードオフ
- アプリケーション層で実装する場合、KVSへのアクセスが発生し、レイテンシが増加する。
- 正規のユーザーでも、短時間にリクエストを繰り返すと制限に引っかかる可能性があり、閾値の設計に運用コストがかかる。

## 適用範囲と例外
- **スコープ:** インターネットに公開されたすべてのAPI、特に認証系や重い集計処理を伴うエンドポイント。
- **例外:** 内部ネットワークで完結する信頼できるマイクロサービス間の通信。

## 検証方法
- k6やJMeterなどの負荷テストツールを使用し、意図的に制限値を超えるバーストトラフィックを送信します。
- レスポンスが `429 Too Many Requests` に切り替わること、および設定した時間経過後に再び `200 OK` に戻ることを確認します。

## 関連ナレッジ
- 外部API通信での一時エラー時に行う指数バックオフとジッター。
- サーキットブレーカー（Circuit Breaker）パターン。

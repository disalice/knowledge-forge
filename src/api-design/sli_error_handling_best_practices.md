## 概要
すべてのAPIにおいて、サービスレベル指標（SLI）を正確に計測するために、REDメトリクス（Rate、Errors、Duration）を統一フォーマットで収集し、エラーハンドリングを明確に分離することが求められます。

## なぜ必要なのか
システムの健全性を「実際のユーザー体験」に基づいて客観的に評価するためには、適切なエラーハンドリングが不可欠です。ステータスコードの適切な分離がなければ、ユーザーの入力ミスによる可用性SLIの低下が発生し、システムの障害検知やエラーバジェットの運用が形骸化してしまいます。

## 実装標準
1. アプリケーションのミドルウェア層で、リクエスト終了時にアクセスログやメトリクスを自動出力する共通基盤を設ける。
2. 出力するメトリクスには以下の情報を含める。
   - タイムスタンプ
   - エンドポイントのルーティングパターン（例: `/users/{id}`）
   - HTTPステータスコード
   - 処理時間（レイテンシ）
3. 5xx系エラーのみを「サーバーの可用性低下」としてカウントする集計ルールを設定する。

## 設計・実装時のチェックリスト
- [ ] バリデーションエラーや認証・認可エラーに対して、誤って500系エラーを返していないか
- [ ] 意図しない例外が確実に500番台として処理・記録される仕組みになっているか
- [ ] メトリクスのラベルに動的なIDが直接記録されていないか（例: `/users/{id}`）

## アンチパターン
- メトリクスのパス情報に動的な値を含め、高カーディナリティ問題を引き起こす。
- エラーの種別を問わず一律でSLI低下として扱い、無用なアラートを頻発させる。
- ビジネスロジックの各所に手動で計測用のコードを埋め込むこと。

## 適用範囲と例外
- ユーザーや外部システムからのリクエストを受け付けるすべてのAPIサーバーに適用。
- ユーザー影響のない非同期のバッチ処理などは別の指標を用いるため、通常のAPI用SLOとは分けて管理。

## 検証方法
- 意図的にバリデーションエラー（400）とシステムエラー（500）を発生させ、ダッシュボード上で500エラーのみが可用性SLIを低下させていることを確認する。
- パスパラメータを含むAPIをコールし、メトリクス基盤上でパスがテンプレート文字列として集約されているか確認する。

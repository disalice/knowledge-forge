## 概要

APIの設計において、バージョニングは重要な要素です。特に、破壊的変更を管理するためには、メジャーバージョンを明記し、既存のエンドポイントを上書きせずに新しいエンドポイントを提供することが求められます。これにより、クライアント側のアップデートタイミングを考慮しつつ、システムの安定性を保つことができます。

## なぜ必要なのか

モバイルアプリや外部システムなど、クライアント側のアップデートをバックエンド側でコントロールできない環境では、既存のシステムを壊すことなく新仕様をリリースすることが重要です。これにより、ユーザーに対する影響を最小限に抑えつつ、APIの進化を図ることができます。

## 実装標準

1. **URIパスベースの採用:** APIエンドポイントにメジャーバージョンを含める（例: `/api/v1/users`）。
2. **バージョンの切り上げ基準:** 
   - 後方互換性のある変更（新フィールド追加など）はバージョンを上げない。
   - 破壊的変更（型の変更、フィールド削除など）の場合のみ新しいメジャーバージョンを作成する。
3. **Sunsetの計画:** 旧バージョンの非推奨化から廃止までの期間を定め、クライアントに告知する。

## 設計・実装時のチェックリスト

- [ ] API仕様の変更が「破壊的変更」か「非破壊的変更」かを明確に判断できているか？
- [ ] 旧バージョンの廃止を予定している場合、クライアント開発者に伝える仕組みが準備されているか？
- [ ] OpenAPI（Swagger）等のドキュメントで、バージョンごとの仕様が明確に分かれて出力されるようになっているか？

## アンチパターン

- **マイナーバージョンのURIへの埋め込み:** `/v1.2/users` のように細かくバージョンを切りすぎてしまい、エンドポイントが乱立する。
- **バージョンなしでの破壊的変更:** 推測だけでレスポンスの型やキー名を変更し、クライアントをクラッシュさせる。

## 実施・導入によるトレードオフ

旧バージョンと新バージョンのコードを同時に維持・保守する必要があり、バックエンドのルーティングやコントローラーのコード量が増加します。また、データベースのスキーマ変更が生じた場合、両方のAPIから正常にデータの読み書きができるようにするため、設計やマイグレーションの難易度が上がります。

## 適用範囲と例外

- **スコープ:** Webフロントエンド、モバイルアプリ、サードパーティの開発者へ公開するすべてのRESTful / HTTP API。
- **例外:** バックエンドと同時にデプロイされることが保証されている内部のマイクロサービス間通信やSSR専用の内部API。

## 検証方法

旧バージョンと新バージョンのそれぞれのエンドポイントに対して統合テストを実施し、単一のデータベースに対してどちらのバージョンからアクセスしても仕様通りに動作し続けることをCIパイプラインで確認します。

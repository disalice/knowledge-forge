## 概要
APIにおけるサービスレベル指標（SLI）を正確に計測するために、すべてのAPIでREDメトリクス（Rate, Errors, Duration）を統一フォーマットで収集し、エラーハンドリングを適切に行うことが求められます。

## なぜ必要なのか
システムの健全性を「実際のユーザー体験」に基づいて客観的に評価するためには、エラーの種類を明確に分けることが重要です。適切に分離されていないステータスコードは、ユーザーの入力ミスによる可用性SLIの低下を招き、システムの障害検知やエラーバジェットの運用に悪影響を及ぼします。

## 実装標準
1. アプリケーションのミドルウェアまたはインターセプター層で、リクエスト終了時にアクセスログやメトリクスを自動出力する共通基盤を設ける。
2. 出力するメトリクスには以下の情報を含める。
   - タイムスタンプ
   - エンドポイントのルーティングパターン（例: `/users/{id}`）
   - HTTPステータスコード
   - 処理時間（レイテンシ）
3. 5xx系エラーのみを「サーバーの可用性低下（SLIの失敗）」としてカウントする集計ルールをダッシュボード側に設定する。

## 設計・実装時のチェックリスト
- [ ] バリデーションエラーや認証・認可エラーに対して、誤って500系エラーを返していないか
- [ ] 意図しない例外（未捕捉のException）が、確実に500番台として処理・記録される仕組みになっているか
- [ ] メトリクスのラベルに動的なIDが直接記録されていないか（例: `/users/{id}` のように丸めること）

## アンチパターン
- メトリクスのパス情報に動的な値を含め、高カーディナリティ問題を引き起こす。
- エラーの種別を問わず一律でSLI低下として扱い、無用なアラートを頻発させる。
- ビジネスロジックの各所に手動で計測用のコードを埋め込む（共通化されていない）。

## 適用範囲と例外
- ユーザーや外部システムからのリクエストを受け付けるすべてのAPIサーバーに適用する。
- ユーザー影響のない非同期のバッチ処理は、別の指標で管理する。

## 検証方法
- 意図的にバリデーションエラー（400）とシステムエラー（500）を発生させ、ダッシュボード上で500エラーのみが可用性SLIを低下させていることを確認する。
- パスパラメータを含むAPIをコールし、メトリクス基盤上でパスがテンプレート文字列として集約されているか確認する。

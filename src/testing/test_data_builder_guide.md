## 概要
テストコード内で複雑なエンティティや集約（Aggregate）のインスタンスを生成する際、コンストラクタを直接呼び出すのではなく、有効なデフォルト値を持った「TestData Builder（テストデータビルダ）」クラスやファクトリライブラリを経由して生成することを標準とします。このアプローチは、特にオブジェクト指向プログラミングやテスト駆動開発（TDD）において、テストの可読性と保守性を向上させるために重要です。

## なぜ必要なのか
- **変更への耐性（Shotgun Surgeryの防止）:** ドメインモデルが成長し、エンティティに新しい必須フィールドが追加された際、コンストラクタを直接呼んでいると数百箇所のテストコードがコンパイルエラーになり、修正作業に忙殺されるのを防ぎます。ビルダーを使えば、ビルダー内のデフォルト値を1箇所修正するだけで済みます。
- **テストの意図（可読性）の明確化:** テストコードをパッと見たとき、「システムの制約を満たすためのどうでもいいダミー値」と「そのテストケースにおいて重要な意味を持つ値」が混在して読みにくくなるのを防ぎます。

## 実装標準
言語やフレームワークに応じたアプローチ（手動実装、または専用ライブラリの利用）を採用します。

1. **ビルダーの作成:** テスト対象のエンティティごとに `UserBuilder` や `OrderBuilder` を作成する。
2. **デフォルト値の保持:** ビルダー内部で、システム的に「正常」とみなされるデフォルトの値を初期状態で持たせておく。
3. **メソッドチェーンによる上書き（Fluent Interface）:** テストシナリオにおいて「関心のある（検証したい）値のみ」を上書きできるようにする。

```typescript
// テストコードでの使用例
// 「18歳未満はエラーになる」というテストの場合、年齢以外はどうでもいい値でよい
const user = new UserBuilder()
  .withAge(17) // このテストで重要なのはここだけ！
  .build();
```

## 設計・実装時のチェックリスト
- [ ] テストコードの `Arrange（準備）` フェーズにおいて、そのテストの目的に無関係なダミー値（例: `"dummy"`, `999`など）が大量に直書きされていないか？
- [ ] エンティティのコンストラクタ仕様が変更された際、無関係なテストケースが壊れない（ビルダー内の修正のみで済む）設計になっているか？

## アンチパターン
- **Object Motherパターンの乱用:** `createNormalUser()`, `createPremiumUser()`, `createPremiumUserWithNoPoints()` のように、固定パターンの生成メソッド（ファクトリメソッド）を無限に作り続けてしまい、最終的に誰も全容を把握できなくなる状態。
- **テスト内でのJSONべた書き:** 各テストケースの中で、数十行かけて巨大なオブジェクトツリーやJSONを手動で組み立てている状態（コードの大部分がテストデータの準備で埋め尽くされてしまう）。

## 実施・導入によるトレードオフ
- 専用ライブラリを使わない場合、各エンティティに対してビルダークラスを作成・保守するための初期実装コストとコード量が増加します。

## 適用範囲と例外
- **スコープ:** ドメイン層（エンティティ、値オブジェクト、集約）およびユースケース層のテストにおけるデータセットアップ。
- **例外:** 引数が1〜2個しかない非常にシンプルな値オブジェクトのテストなど、ビルダーを用意するオーバーヘッドが明確に無駄になるケース。

## 検証方法
- テストコード（特に準備フェーズ）を読む際、`build()` メソッドの直前にチェーンされている `withXxx()` を見るだけで、「このテストケースはどの条件・エッジケースを検証しようとしているのか」が瞬時に理解できるかをレビューする。

## 関連ナレッジ
- アーキテクチャ層ごとのテスト責務とモック（Mock/Stub）の境界線
- 集約（Aggregate）の設計と他集約への参照ルール（集約が大きくなると生成が大変になるため、ビルダーの恩恵が最大化されます）

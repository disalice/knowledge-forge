## 概要

クリーンアーキテクチャにおける各レイヤー（ドメイン層、ユースケース層、インフラ層）ごとに自動テストの目的を明確にし、「モック化してよいもの」と「絶対にモック化してはいけないもの」の境界線を標準化します。特に、ドメイン層（エンティティや値オブジェクト）に対するテストではモックの使用を全面禁止とし、実際のインスタンスを用いてテストを行います。

## なぜ必要なのか

- **テストの形骸化（モック地獄）を防ぐため:** ユースケースのテストにおいて、呼ばれるエンティティの振る舞いまでモック化してしまうと、テストが無意味になりバグを検知できなくなるため。
- **リファクタリング耐性の向上:** 内部の実装詳細に依存しない、純粋な「入力と出力」のテストにすることで、コードの構造を変えてもテストが落ちないようにするため。

## 実装標準

1. **ドメイン層（Entities / Value Objects）のテスト:**
   - 外部依存（DBやAPI）を持たないため、モックライブラリの使用は一切禁止します。本物のインスタンスを生成し、状態遷移や計算のロジックが正しいかを高速で検証します。

2. **ユースケース層（Use Cases）のテスト:**
   - テスト対象のユースケースに本物のエンティティを渡します。ただし、データベースや外部APIと通信する「Repositoryインターフェース」のみをモック（またはインメモリのFake実装）に差し替えます。

3. **インフラ層（Repositories / Controllers）のテスト:**
   - 正しくSQLが発行されるか、APIのJSONスキーマが合っているかを検証します。モックは使わず、テスト用のローカルデータベースやスタブサーバーを立ち上げ、実際の通信を伴うインテグレーションテストを行います。

## 設計・実装時のチェックリスト

- [ ] ドメイン層のユニットテストコード内に、モックライブラリのインポートが存在していないか？
- [ ] ユースケース層のテストで、エンティティの戻り値まで都合よくモック化（過剰なモック）していないか？
- [ ] DBアクセス（Repository）のテストで、DBへの実際の書き込みを避けるためにクエリビルダそのものをモック化していないか？

## アンチパターン

- **実装の過剰な検証 (Overspecification):** 内部の挙動を細かくテストしすぎる状態。コードを少し整理しただけでテストが壊れます。
- **Ice Cream Cone（逆ピラミッド）:** モックを使わずに全てを結合した重たいテストが肥大化し、ドメインの単体テストが薄い状態。

## 実施・導入によるトレードオフ

- 依存関係を適切に切り離す設計になっていないと、このルールに従ってテストを書くことが困難になります。
- インフラ層のテストにおいて、テスト用データベースの起動とマイグレーション実行というオーバーヘッドが発生します。

## 適用範囲と例外

- **スコープ:** 新規実装およびリファクタリングされるバックエンドのすべての自動テスト。
- **例外:** 現在の日時や乱数に依存するドメインロジックのテストを行う場合のみ、ドメイン層であっても時刻や乱数生成器のインターフェースを固定化することを許可します。

## 検証方法

- テストカバレッジレポートを確認し、特にドメイン層のブランチカバレッジが高く、次にユースケース層が高くなっている「正しいテストピラミッド」の形になっているかを確認します。
